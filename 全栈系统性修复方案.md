# ITops Agent 全栈系统性修复方案

> **生成时间**：2026-07-02
> **目标**：系统性修复"约束形同虚设"问题，让每一层防护真正生效
> **执行方**：AI 自动修复 + 人工验收
> **核心原则**：先修复防护层（让约束长牙齿），再逐层清理技术债

## 一、问题总览

项目设计了完整的多层防护体系（DI 容器、Repository 模式、ESLint 规则、CI 流水线、命令过滤器、加密服务），但**每一层约束在落地时都被系统性绕过**，导致问题持续累积且无法被自动发现。

| 优先级 | 类别 | 问题数 | 修复目标 |
|---|---|---|---|
| **P0** | 安全漏洞（可被直接利用） | 5 | 消除立即爆雷风险 |
| **P1** | CI/CD 质量门禁失效 | 6 | 让其他问题能被自动拦截 |
| **P2** | 后端架构约束被绕过 | 6 | 让 DI/Repository/分层真正生效 |
| **P3** | 前端工程化缺失 | 7 | 建立前端 Service 层和类型安全 |
| **P4** | 渐进式技术债 | 5 | 持续优化 |

### P0-1：命令过滤器黑名单可绕过 → 远程命令注入

**问题**：[commandFilter.ts:14](file:///f:/自开发代码/多Agent运维平台/ITops%20agent/daima/backend/src/middleware/commandFilter.ts#L14) 用 `^rm\s+-rf\s+\/{1,3}$` 正则锚定完整命令行；`echo x; rm -rf /` 即可绕过。命令最终通过 [sshCommandExecutor.ts:57](file:///f:/自开发代码/多Agent运维平台/ITops%20agent/daima/backend/src/modules/servers/services/sshService/sshCommandExecutor.ts#L57) `conn.exec(command)` 在远程服务器执行。

**修复步骤**：

1. 将 `commandFilter.ts` 从"黑名单匹配完整命令"改为"命令解析 + 危险操作检测"：
   ```typescript
   // 用 shell-quote 解析命令为 token 数组
   import { parse as parseShell } from 'shell-quote';
   
   export function checkCommandSafety(command: string, userRole: string) {
     // 1. 检测命令链操作符（; && || |）后是否跟危险命令
     const tokens = parseShell(command);
     const segments = splitByOperators(tokens); // 按 ; && || | 分段
     
     for (const seg of segments) {
       const cmd = seg[0]?.toString() || '';
       // 2. 对每段独立检查
       if (DANGEROUS_COMMANDS.has(cmd)) { ... }
     }
     
     // 3. 检测子 shell $(...) 和反引号
     if (/\$\(|`/.test(command)) {
       return { allowed: false, severity: 'blocked', reason: '子 shell 执行被禁止' };
     }
     
     // 4. 检测 base64/hex 解码管道
     if (/base64\s+-d\s*\|/.test(command) || /xxd\s+-r/.test(command)) {
       return { allowed: false, severity: 'blocked', reason: '编码解码管道被禁止' };
     }
   }
   ```

2. 对于交互终端（`terminalService.ts` 的 `sendData`），改为会话级审计模式：
   - 不尝试逐字符过滤（不可能有效）
   - 记录完整命令历史到 `command_audit_logs` 表
   - 对检测到的危险命令发出实时告警（不阻断但记录）

3. 安装依赖：`npm install shell-quote`

**验收标准**：
- [ ] `echo x; rm -rf /` 被拦截
- [ ] `$(rm -rf /)` 被拦截
- [ ] `echo cm0gLXJmIC8= | base64 -d | sh` 被拦截
- [ ] 交互终端命令审计生效
- [ ] 新增命令过滤单测覆盖所有绕过场景

### P0-2：加密主密钥存储在数据库中，与密文同库

**问题**：[encryptionService.ts:13-31](file:///f:/自开发代码/多Agent运维平台/ITops%20agent/daima/backend/src/modules/auth/services/encryptionService.ts#L13) 将 AES-256 主密钥以 base64 明文存于 `encryption_keys` 表，与加密后的服务器密码/SSH 私钥位于同一 SQLite 文件。数据库泄露即全部凭据泄露。

**修复步骤**：

   ```typescript
   function deriveEncryptionKey(): Buffer {
     const secret = env.JWT_SECRET;
     // 使用 PBKDF2 派生密钥，与 credentialService 一致
     return crypto.pbkdf2Sync(secret, 'encryption-service-v2', 100000, KEY_LENGTH, 'sha512');
   }
   ```

2. 实现密钥迁移：

   - 用旧密钥解密所有 `servers.password` 和 `servers.private_key`
   - 用新派生密钥重新加密
   - 迁移完成后删除 `encryption_keys` 表中的密钥记录（保留表结构供回滚）
   - 添加 `key_version` 列到 `servers` 表区分 v1/v2 密文

3. 数据迁移 migration：新建 `v056_encryption_key_migration.ts`

**验收标准**：
- [ ] `encryption_keys` 表不再存储实际密钥值
- [ ] 密钥从 `JWT_SECRET` 派生，不落盘
- [ ] 已有服务器凭据自动迁移到新密钥
- [ ] 迁移后旧密文可正确解密

### P0-5：Webhook 签名对重新序列化 body 计算

**问题**：[webhookRoutes.ts:165](file:///f:/自开发代码/多Agent运维平台/ITops%20agent/daima/backend/src/modules/infra/routes/webhookRoutes.ts#L165) `JSON.stringify(req.body)` 将 Express 解析后的对象重新序列化为字符串再计算 HMAC，与标准签名实践不一致。`warn` 模式下缺失签名的请求被直接放行。

**修复步骤**：

1. 添加 `express.raw()` 中间件捕获原始 body：
   ```typescript
   // 在 JSON 解析之前注册 raw body 捕获
   import { Router } from 'express';
   
   const router = Router();
   
   // 捕获原始 body 字节流用于签名验证
   router.use('/webhooks/:source', express.json({
     verify: (req, _res, buf) => {
       (req as any).rawBody = buf;
     }
   }));
   ```

2. 修改签名验证使用原始 body：
   ```typescript
   function verifyWebhookSignature(req: Request, secret: string): boolean {
     const signature = req.headers['x-signature'] as string;
     if (!signature) return false;
     
     // 使用原始 body 字节流
     const rawBody = (req as any).rawBody as Buffer;
     if (!rawBody) return false;
     
     const expected = crypto
       .createHmac('sha256', secret)
       .update(rawBody)
       .digest('hex');
     
     // ...timingSafeEqual 比较
   }
   ```

3. 在 `warn` 模式下增加告警计数和暴露：
   - 记录到 `webhook_security_events` 表
   - 在 `/health` 接口暴露未签名请求数

**验收标准**：
- [ ] 使用原始 body 字节流计算签名
