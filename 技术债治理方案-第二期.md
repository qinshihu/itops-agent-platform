## 二、总体架构目标

```
Route Handler (仅参数校验 + 调用 Service)
       ↓
Service Layer (业务逻辑，通过 DI 容器获取 Repository)
       ↓
Repository Layer (数据访问，唯一允许 import db 的地方)
       ↓
Database (better-sqlite3)
```

**核心原则：**
1. `src/repositories/` 是唯一允许 `import db from`的业务代码目录
2. Route Handler 不允许直接 `import db`，只能调用 Service
3. Service 通过 `container.get('xxxRepository')` 或直接 `import { xxxRepository }` 获取 Repository
4. 新 Repository 必须注册到 DI 容器，支持 `container.replace()` 测试替换
5. 每次迁移保持 `npm run build` + `npm test` 全绿

### 阶段 5：导入风格统一与 DI 依赖声明（预计 2 天）

#### 5.1 统一 db 导入风格

将 3 种导入风格统一为**默认导入**：
- `import { db } from` → `import db from`（15 个文件）
- `import db, { xxx } from` → `import db from` + 从对应模块 import { xxx }（5 个文件）

containers/kubernetes/dc 三个模块全部需要改。

#### 5.2 声明 DI 依赖关系

为 `serviceRegistry.ts` 中 24 个服务补充 `dependencies` 声明：

```typescript
// 现状（无依赖声明）
container.register('alertService', () => new AlertService(), []);

// 改进后
container.register('alertService', 
  () => new AlertService(container.get('alertRepository'), container.get('serverRepository')),
  ['alertRepository', 'serverRepository']
);
```

使拓扑排序真正生效，循环依赖在启动时即被检测。

#### 5.3 逐步消除 `import db from` 的非 repository 文件

在 `tsconfig.json` 或 ESLint 中增加路径限制规则：
- `src/modules/` 下不允许 `import db from '../../models/database'`
- 只允许 `import { xxxRepository } from '../../repositories'`
- 或 `import { container } from '../../core/serviceContainer'`

**阶段 5 总验收：**
- `import db from` 仅存在于 `src/repositories/` 和 `src/models/` 目录
- DI 容器 24 个服务均声明 dependencies
- ESLint 规则阻止 modules/ 下的新 db 导入

| 导入风格统一破坏 containers 模块 | 低 | 先统一导入风格再做内容迁移；逐文件改并验证 build |

## 八、附录：迁移单个文件的 Checklist

每次将一个文件从 `import db` 迁移到 Repository 时，按以下步骤：

- [ ] 识别该文件中所有 `db.prepare()` 调用
- [ ] 将每个调用映射到对应 Repository 方法（如无对应方法则先在 Repository 中新增）
- [ ] 替换 `import db from` 为 `import { xxxRepository } from '../../repositories'`
- [ ] 验证 `npm run build` 通过
- [ ] 运行相关测试（`npm test -- --reporter=verbose xxx`）
- [ ] 检查是否有遗漏的 `db.prepare` / `db.exec` / `db.transaction` 调用
- [ ] 确认文件中不再有任何 `db` 引用
- [ ] 运行全量测试 `npm test`
