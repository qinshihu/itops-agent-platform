# daima 项目技术债治理方案

> **文档目的**：为 AI 编程工具提供可执行的分阶段重构方案，逐步消除 daima 项目的架构漂移问题。
> **项目路径**：`f:\自开发代码\多Agent运维平台\ITops agent\daima`
> **后端路径**：`f:\自开发代码\多Agent运维平台\ITops agent\daima\backend`
> **创建日期**：2026-07-01
> **执行原则**：每阶段独立可上线、独立可回滚；先对齐再删除；小步快跑。

---

## 一、现状诊断

### 1.1 项目概况

daima 是「AI 多 Agent 协作的企业级运维自动化平台」(v3.0.5)，技术栈：Node.js 22 + TypeScript + Express + SQLite(better-sqlite3) + React 18。后端 12 个业务模块，41 个 DB migration 版本。

### 1.2 核心问题清单

| # | 问题 | 严重度 | 证据 |
|---|------|--------|------|
| 1 | 两套多 Agent 实现并存，未完成切换 | ⚠️ 高 | 旧版 `MultiAgentOrchestrator` 与新版 `Coordinator+Specialist` 同时存在 |
| 2 | LLM 调用硬编码豆包，绕过模型池抽象 | ⚠️ 高 | 5 个文件直接 `callDoubaoAPI`，用户切换模型不生效 |
| 3 | `ensureTables()` 在运行时建表，绕过 migration | ⚠️ 中高 | 9 个 service 在启动时 `CREATE TABLE IF NOT EXISTS` |
| 4 | 246 处 `db.exec/prepare/run` 散落业务代码，无 Repository 层 | ⚠️ 中 | 测试无法 mock db，测试覆盖低 |
| 5 | ServiceContainer 形同虚设 | ⚠️ 中 | 容器存在但业务代码直接 `import db` |
| 6 | 巨型文件累积（72KB/49KB/44KB/37KB） | ⚠️ 中 | 单文件违反单一职责 |
| 7 | 测试覆盖断层（36 测试文件 vs 数百源文件） | ⚠️ 中 | AARS 核心编排器 0 测试 |

### 1.3 根因

项目经历了至少一次大型架构升级（旧 MultiAgentOrchestrator → 新 Coordinator/Specialist、直接 import → ServiceContainer、硬编码 LLM → Provider 抽象），但每次升级只完成 80%，旧代码未删除，新代码未完全接管主路径。

---

## 二、总体策略

5 个阶段、由低风险到高风险推进：

```
阶段1 (1-2天) │ 多Agent系统合并           ← 风险最低，收益最大，立即做
阶段2 (2-3天) │ LLM调用统一到generateCompletion
阶段3 (3-5天) │ ensureTables下沉到migration  ← 涉及生产schema，谨慎
阶段4 (3-5天) │ Repository层抽离 + 巨型文件拆分
阶段5 (持续)  │ 测试补齐 + DI容器真正启用
```

**依赖关系**：阶段 1、2 互相独立可并行；阶段 3 独立；阶段 4 依赖阶段 1 完成（避免改到已删除代码）；阶段 5 依赖阶段 4 的 Repository 层。

## 三、阶段 1：多 Agent 系统合并（最高优先级）

### 3.1 现状

- **新版（保留）**：`backend/src/modules/ai/services/multiAgent/Coordinator.ts` + `Specialists.ts`，被 `backend/src/modules/ai/routes/multiAgentRoutes.ts` 的 `/api/multi-agent/task` 等端点使用，是活跃生产路径
- **旧版（删除）**：`backend/src/modules/ai/services/agents/multiAgentCollaboration.ts` 的 `MultiAgentOrchestrator`，仅被 `services/index.ts` re-export 和 `tests/core.test.ts` 引用

### 3.2 步骤

#### 步骤 1.1：确认旧版无其他调用方

执行命令：
```bash
cd backend
grep -rn "MultiAgentOrchestrator\|AgentMessageBus" src --include="*.ts" --exclude="*.test.ts" --exclude="multiAgentCollaboration.ts"
```

预期输出仅 `src/modules/ai/services/index.ts` 的 re-export 行。若有其他调用方，先逐一迁移到 `getCoordinator().executeTask()`。

#### 步骤 1.2：改造 `tests/core.test.ts`

文件：`backend/src/tests/core.test.ts` 第 72-80 行

将：
```typescript
const { MultiAgentOrchestrator } = await import('../modules/ai/services/agents/multiAgentCollaboration');
const orchestrator = new MultiAgentOrchestrator('test-task');
```

改为：
```typescript
const { executeTask, initializeMultiAgentSystem } = await import('../modules/ai/services/multiAgent');
await initializeMultiAgentSystem();
const result = await executeTask('test-task-input');
```

若测试断言依赖旧版 API 形状（如 `routeToBestAgent`），改为调用 `specialistRegistry.selectBestSpecialistForTask()`。

#### 步骤 1.3：清理 `services/index.ts` 的 re-export

文件：`backend/src/modules/ai/services/index.ts` 第 28-35 行

删除：
```typescript
export {
  MultiAgentOrchestrator,
  AgentMessageBus,
} from './agents/multiAgentCollaboration';
export type {
  CollaborationMessage,
  AgentCollaborationContext,
} from './agents/multiAgentCollaboration';
```

补上新版导出：
```typescript
// ── 多 Agent 协作（Coordinator+Specialist 双层架构）──
export {
  initializeMultiAgentSystem,
  getCoordinator,
  executeTask,
  specialistRegistry,
} from './multiAgent';
export type { Task, TaskResult, Specialist } from './multiAgent';
```

#### 步骤 1.4：删除旧文件

- 删除 `backend/src/modules/ai/services/agents/multiAgentCollaboration.ts`
- 删除 `backend/src/modules/ai/services/agents/multiAgentCollaboration.test.ts`

#### 步骤 1.5：验收

```bash
cd backend
npm run build           # 必须通过
npm test                # 必须全绿
grep -rn "MultiAgentOrchestrator" src   # 必须无结果
```

手动测试：调用 `POST /api/multi-agent/task` 接口，确认返回正常。

### 3.3 风险与回滚

- **风险**：低。新版已是活跃路径，旧版仅测试引用
- **回滚**：`git revert` 即可

## 四、阶段 2：LLM 调用统一到 `generateCompletion`

### 4.1 现状

- 项目已有完整模型池管理：`backend/src/modules/ai/services/models/aiModelService.ts`（`getEnabledModels`/`getDefaultModel`/`provider_type` 抽象）
- `backend/src/modules/ai/services/llm/llmService.ts` 中 `generateCompletion` 是统一入口，已被 RCA/SSH/网络/告警等约一半代码使用
- 但 5 个文件仍硬编码 `callDoubaoAPI`：

| 文件 | 硬编码调用 |
|------|-----------|
| `services/multiAgent/Coordinator.ts` | `callDoubaoAPI` |
| `services/multiAgent/Specialists.ts` | `callDoubaoAPI` |
| `services/agents/copilotService.ts` | `callDoubaoAPI, callOpenAIAPI, callLocalAIAPI` |
| `services/agents/agentExecutor.ts` | `callDoubaoAPIWithTools` |
| `services/agents/multiAgentCollaboration.ts` | （阶段1已删除） |

### 4.2 步骤

#### 步骤 2.1（前置）：验证 `generateCompletion` 是否真的走模型池

读 `backend/src/modules/ai/services/llm/llmService.ts` 中 `generateCompletion` 函数实现，确认：
1. 内部是否调用 `getDefaultModel()` 或 `getEnabledModels()`
2. 是否按 `model.provider_type` 分发到不同厂商 API

**若 `generateCompletion` 仍硬编码豆包**：先改造它，让它走 `aiModelService.getDefaultModel()` + 按 `provider_type` 分发。这是阶段 2 的前置条件，不满足则暂停后续步骤。

#### 步骤 2.2：改造 `Coordinator.ts`

文件：`backend/src/modules/ai/services/multiAgent/Coordinator.ts`

将：
```typescript
import { callDoubaoAPI } from '../llm/llmService';
// ...
const result = await callDoubaoAPI(prompt, systemPrompt, agentName, temperature, signal);
```

改为：
```typescript
import { generateCompletion } from '../llm/llmService';
// ...
const result = await generateCompletion({
  prompt, systemPrompt, agentName, temperature, signal,
});
```

注意：确认 `generateCompletion` 的实际签名（可能不是对象参数，按实际调整）。

#### 步骤 2.3：改造 `Specialists.ts`

文件：`backend/src/modules/ai/services/multiAgent/Specialists.ts`

同步骤 2.2，把所有 `callDoubaoAPI` 调用改为 `generateCompletion`。Specialists.ts 有多个 Specialist 类，逐一改造。

#### 步骤 2.4：改造 `copilotService.ts`

文件：`backend/src/modules/ai/services/agents/copilotService.ts`

把 `callDoubaoAPI, callOpenAIAPI, callLocalAIAPI` 全部改为 `generateCompletion`。

#### 步骤 2.5：改造 `agentExecutor.ts`（最复杂）

文件：`backend/src/modules/ai/services/agents/agentExecutor.ts`

此文件用了 `callDoubaoAPIWithTools`（带工具调用的变体）。处理方案：

**方案 A（推荐）**：若 `generateCompletion` 已支持 `tools` 参数，直接替换
**方案 B**：若 `executeAgentWithLLM`（更高级抽象）已存在且走模型池，改用 `executeAgentWithLLM`
**方案 C**：在 `llmService.ts` 新增 `generateCompletionWithTools()` 函数，内部按模型池分发

#### 步骤 2.6：移除 `callDoubaoAPI` 等的公开导出

文件：`backend/src/modules/ai/services/index.ts` 第 9-21 行

确认无外部引用后：
```bash
grep -rn "callDoubaoAPI\|callOpenAIAPI\|callLocalAIAPI" src --include="*.ts" --exclude="*.test.ts" --exclude="llmService.ts"
```

无结果则从 `services/index.ts` 移除这三个函数的 re-export（保留 `llmService.ts` 内部实现一个版本周期，标注 `@deprecated`）。

### 4.3 验收

```bash
# 1. 构建与测试通过
cd backend && npm run build && npm test

# 2. 无硬编码调用
grep -rn "callDoubaoAPI\|callDeepseekAPI\|callQwenAPI" src --include="*.ts" --exclude="*.test.ts" --exclude="llmService.ts"
# 必须无结果
```

**功能验收**：在 UI 把默认模型切换为 DeepSeek/Ollama，执行一个 Agent 任务，看后端日志确认实际调用了对应 API（而非豆包）。

### 4.4 风险与回滚

- **风险**：中。`generateCompletion` 若未真正走模型池，需先补齐
- **回滚**：保留 `callDoubaoAPI` 函数一个版本周期，标注 `@deprecated`，发现问题可切回

## 五、阶段 3：`ensureTables` 下沉到 migration

### 5.1 现状

9 个 service 有 `ensureTables()` 方法，在 `app.ts`/`serviceRegistry.ts` 启动时统一调用，违反 migration 框架的单一 schema 来源原则。

涉及 service 清单：
1. `modules/alerts/services/alertAutoResponse/alertAutoResponseService.ts`（建 `baseline_metrics`/`aars_response_logs`/`aars_config`）
2. `core/AlertProcessor.ts`
3. `modules/infra/services/composeService.ts`
4. `modules/containers/services/registryService.ts`
5. `modules/containers/services/multiHostDockerService.ts`
6. `modules/kubernetes/services/kubernetesService.ts`
7. `modules/auto/services/autoScaleService.ts`
8. `modules/containers/services/vmMigrationService.ts`
9. `modules/containers/services/vmSnapshotSchedulerService.ts`

### 5.2 步骤

#### 步骤 3.1：全量比对

对每个 service，执行：
1. grep `CREATE TABLE` 提取所有建表语句和表名
2. 在 `backend/src/models/migrations/` 下 grep 确认是否已有同名表
3. 对照字段定义是否一致

预期三种情况：
- **A. migration 已建，ensureTables 冗余**（多数情况，如 AARS 的 `aars_response_logs` 在 v018 已建）→ 直接删除 ensureTables
- **B. migration 未建，ensureTables 是唯一来源** → 新增 migration 版本建表，再删 ensureTables
- **C. 两边定义不一致** → 以 migration 为准，修正后删除 ensureTables

#### 步骤 3.2：分批改造（每 service 一个独立 PR）

**强烈建议**每个 service 单独 PR，避免大爆炸。建议顺序：

1. `alertAutoResponseService`（AARS，最核心，先改）
2. `AlertProcessor`
3. `composeService`、`registryService`、`multiHostDockerService`
4. `kubernetesService`、`autoScaleService`
5. `vmMigrationService`、`vmSnapshotSchedulerService`

#### 步骤 3.3：每个 PR 的标准改动

以 `alertAutoResponseService` 为例：

**a. 若情况 B/C（migration 未建或不一致）**：

新建 `backend/src/models/migrations/v042_alert_auto_response_tables.ts`：
```typescript
import type { Migration } from './migrationFramework';

export const migration: Migration = {
  version: 42,
  name: 'alert_auto_response_tables',
  up: (db) => {
    db.exec(`
      CREATE TABLE IF NOT EXISTS baseline_metrics (
        device_id TEXT NOT NULL,
        metric_name TEXT NOT NULL,
        sample_value REAL NOT NULL,
        sampled_at TEXT NOT NULL DEFAULT (datetime('now','localtime')),
        PRIMARY KEY (device_id, metric_name, sampled_at)
      );
      CREATE TABLE IF NOT EXISTS aars_response_logs (
        -- 完整字段定义，与 ensureTables 中一致
      );
      CREATE TABLE IF NOT EXISTS aars_config (
        -- 完整字段定义
      );
    `);
  },
  down: (db) => {
    // 可选：drop 三张表
  },
};
```

在 `backend/src/models/migrations/index.ts` 注册：
```typescript
import { migration as v042 } from './v042_alert_auto_response_tables';
// ...
export const migrations = [/* ... */, v041, v042];
```

**b. 删除 service 的 `ensureTables()` 方法**

在 `alertAutoResponseService.ts` 中删除 `private ensureTables()` 方法及其调用（构造函数中的 `this.ensureTables()`）。

**c. 删除 serviceRegistry 中的调用**

文件：`backend/src/serviceRegistry.ts`（约第 192-198 行）

删除对应 service 的 `xxxService.ensureTables()` 调用。

#### 步骤 3.4：验收

每个 PR 验收：

```bash
cd backend
npm run build && npm test

# 全新 SQLite 文件测试
rm test.db
npm run migrate           # 跑所有 migration
sqlite3 test.db ".dump" > after.sql

# 与改造前对比（需提前保存 before.sql）
diff before.sql after.sql  # 表结构应一致
```

最终目标：`grep -rn "ensureTables" backend/src` 无结果。

### 5.3 风险与回滚

- **风险**：中高。涉及生产 schema
- **缓解**：每个 service 单独 PR；先在测试环境跑一周；migration 用 `CREATE TABLE IF NOT EXISTS` 不破坏已有数据
- **回滚**：migration 是增量的，新版本不破坏旧表；ensureTables 删除可单独 revert

## 六、阶段 4：Repository 层抽离 + 巨型文件拆分

### 6.1 子阶段 4A：Repository 层抽离

#### 现状

246 处 `db.exec/prepare/run` 散落 30+ 文件，业务代码直接 `import db from '../../../../models/database'`，测试无法 mock。

#### 步骤

**步骤 4A.1：建立 repository 目录结构**

新建 `backend/src/repositories/`，按模块组织：
```
backend/src/repositories/
├── alertRepository.ts
├── agentExecutionRepository.ts
├── dcRepository.ts
├── networkRepository.ts
├── ...
└── index.ts   (barrel file)
```

**步骤 4A.2：封装优先级最高的 3 个 repository**

按 SQL 调用频率排序，优先：

1. **`alertRepository.ts`**（alerts 模块 30+ 处 SQL）
   - 封装 `getAll`/`getById`/`create`/`update`/`delete`/`acknowledge`/`resolve` 等
   - 业务代码改为 `import { alertRepository } from '../../repositories/alertRepository'`

2. **`agentExecutionRepository.ts`**（multiAgentRoutes 等多处 INSERT agent_executions）
   - 封装 `createExecution`/`getExecution`/`updateStatus`

3. **`dcRepository.ts`**（dc 模块 60+ 处 SQL）
   - 按 entity 拆：`rackRepo`/`deviceRepo`/`pduRepo`/`cableRepo`

**步骤 4A.3：repository 接口规范**

每个 repository 遵循：
```typescript
// backend/src/repositories/alertRepository.ts
import db from '../models/database';
import type { Alert } from '../types/alert';

export const alertRepository = {
  getAll(filters: AlertFilters): Alert[] {
    return db.prepare('SELECT * FROM alerts WHERE ...').all(filters);
  },
  getById(id: string): Alert | undefined {
    return db.prepare('SELECT * FROM alerts WHERE id = ?').get(id);
  },
  // ...
};
```

**步骤 4A.4：渐进式迁移业务代码**

不必一次性全改。新代码强制用 repository，老代码逐步迁移。每改一个文件，删掉其 `import db`。

#### 验收

```bash
# 业务代码不再直接 import db
grep -rn "import db from" backend/src/modules --include="*.ts" --exclude="*.test.ts"
# 数量应从 30+ 逐步降到 0
```

### 6.2 子阶段 4B：巨型文件拆分

#### 拆分清单

| 文件 | 大小 | 拆分方案 |
|------|------|----------|
| `modules/network/services/vendorAdapter.ts` | 72KB | 拆为 `vendorAdapter/index.ts` + 每厂商一个文件：`cisco.ts`/`huawei.ts`/`h3c.ts`/`juniper.ts`/... |
| `services/mcp/toolDefinitions.ts` | 49KB | 按域拆分：`networkTools.ts`/`containerTools.ts`/`serverTools.ts`/`dcTools.ts`/`alertTools.ts` |
| `modules/workflow/services/workflowExecutor.ts` | 44KB | 抽离 `nodeExecutor.ts`/`stateMachine.ts`/`contextManager.ts` |
| `modules/ai/services/llm/llmService.ts` | 37KB | 拆为 `llmService/index.ts` + `circuitBreaker.ts` + `providerAdapters.ts` + `toolCalling.ts` |
| `modules/containers/services/vmManagement/proxmoxAdapter.ts` | 37KB | 按 resource 拆：`vm.ts`/`storage.ts`/`network.ts`/`node.ts` |
| `modules/containers/services/vmManagement/kvmAdapter.ts` | 36KB | 同上 |
| `modules/alerts/services/alertAutoAnalyzer.ts` | 34KB | 抽离 `noiseReducer.ts`/`correlator.ts`/`enricher.ts` |
| `models/migrations/v001_initial_schema.ts` | 35KB | **不动**（历史快照，保留原样） |

#### 拆分原则

1. 先建新文件，移动代码
2. 老文件改为 barrel：`export * from './submodule'`
3. 保持对外导出兼容，外部引用无需改动
4. 逐步更新引用到新路径（可选优化）

### 6.3 风险与回滚

- **风险**：中。主要是 import 路径变更可能漏改
- **回滚**：barrel file 保证向后兼容，可随时 revert 拆分

## 七、阶段 5：测试补齐 + DI 容器启用

### 7.1 子阶段 5A：测试补齐

#### 优先级（按"风险×变更频率"排序）

1. `modules/alerts/services/alertAutoResponse/alertAutoResponseService.ts`（核心编排器，0 测试）
2. `modules/ai/services/multiAgent/Coordinator.ts` + `Specialists.ts`（多 Agent 核心）
3. `modules/workflow/services/workflowExecutor.ts`（工作流引擎）
4. 各 repository（阶段 4A 产出后，测试自然聚集到 repository 层）

#### 目标

关键路径行覆盖率达到 60%，先求有再求精。

### 7.2 子阶段 5B：DI 容器真正启用

#### 现状

`backend/src/core/serviceContainer.ts` 已有完整实现（拓扑排序、生命周期钩子），但业务代码不用它，直接 `import db`。

#### 步骤

1. 阶段 4A 产出的 repository 先注册到 container：
   ```typescript
   // backend/src/serviceRegistry.ts
   container.register('alertRepository', () => alertRepository);
   container.register('agentExecutionRepository', () => agentExecutionRepository);
   ```

2. 新代码强制通过 `container.get('alertRepository')` 获取

3. 老代码不强制改，自然迁移

4. 最终目标：测试时 `container.register('alertRepository', () => mockRepo)` 即可替换

## 八、执行节奏建议

| 阶段 | 工作量 | 建议节奏 | 可并行 |
|------|--------|----------|--------|
| 1 | 1-2天 | 单 PR，立即做 | 否 |
| 2 | 2-3天 | 5 个子 PR，逐文件 | 部分可并行 |
| 3 | 3-5天 | 9 个子 PR，每 service 一个 | **可并行**（多人协作） |
| 4A | 3-5天 | 3-5 个 repository PR | **可并行** |
| 4B | 3-5天 | 5 个拆分 PR | **可并行** |
| 5 | 持续 | 随功能开发持续补 | 持续 |

**关键原则**：
- 每阶段独立可上线，不必等全部完成
- 每阶段做完跑一次完整回归（启动 + 主要 API + 关键工作流）
- 阶段 1、2 收益最大风险最低，建议本周内完成
- 阶段 3 涉及生产 schema，建议先在测试环境跑一周
