# ADR-005: 模块化领域拆分（DDD 限界上下文）

**状态**: Superseded by ADR-017 | **日期**: 2025-Q1 | **决策者**: 项目作者

> **更新 (2026-07-07, ADR-017)**：本 ADR 原定 **18 个** 模块，P1-6 阶段 `infra/` 模块按业务子域拆分（详见 [ADR-017](./017-infra-subdomain-splitting.md)），现 backend 已扩展至 **24 个模块**、frontend 至 **23 个模块**。本 ADR 仍保留作为历史决策，模块表清单以 [`rules/architecture.md §1.2`](../rules/architecture.md) 为准。
>
> **现状提示**：本 ADR 主要保留**历史决策轨迹**。具体的模块清单、约束规则、聚合根保护、统一领域语言等内容，请直接查阅 [`rules/architecture.md §1.2 / §二`](../rules/architecture.md)（权威源）；本 ADR 仅作背景阅读。

## 背景

项目早期代码按功能散落，没有明确的模块边界。AI 辅助开发时经常出现跨模块乱改（为了省事，把告警规则写在服务器管理接口里、把配置解析逻辑放在数据库操作里）。

核心需求：
1. 严格模块边界，AI 只能在限界上下文内操作
2. 模块间通过明确接口通信
3. 每个模块有清晰的职责说明
4. 新增功能自动归位到正确的模块

## 决策

将系统拆分为 **18 个独立模块**，每个模块对应一个 DDD 限界上下文。

## 模块划分

| 模块 | 领域职责 | 为什么独立 |
|------|---------|-----------|
| `ai/` | AI 能力编排 | LLM 调用、Agent 管理、RCA 是独立领域 |
| `alerts/` | 告警全生命周期 | 告警是独立事件流，有自己的规则引擎 |
| `auth/` | 认证与授权 | 安全边界必须独立 |
| `auto/` | 自动化策略 | 修复策略和伸缩策略和告警是不同的关注点 |
| `backup/` | 数据库备份恢复 | 备份加密、定时策略与基础设施分离 |
| `change-management/` | 变更管理 | 变更审批流转是独立流程 |
| `config-management/` | 配置管理 | 配置模板、配置修复、Docker Compose 独立编排 |
| `containers/` | 容器与 VM | Docker/K8s/VM 资源管理与服务器管理分离 |
| `database/` | 数据库连接 | 独立的连接管理域 |
| `dc/` | 数据中心设备 | 机房、机架、电源是独立物理域 |
| `infra/` | 系统基础设施 | 脚本管理、审计日志、工具链接、系统设置 |
| `kubernetes/` | K8s 管理 | 与 Docker 不同的资源编排模型 |
| `mcp/` | MCP 工具协议管理 | 工具注册、JSON-RPC 网关、外部 Server 管理是独立协议层 |
| `monitor/` | 监控与报告 | 只读分析域，与操作域分离 |
| `network/` | 网络设备 | 21+ 厂商适配器，独立领域 |
| `notification/` | 通知管理 | 多渠道通知与基础设施分离 |
| `servers/` | 服务器管理 | SSH/密钥/远程桌面，独立生命周期 |
| `workflow/` | 工作流引擎 | 编排引擎是核心基础设施 |

## 模块间通信规则

```
✅ 允许：Module A/services/ → Module B/services/
✅ 允许：Module A/services/ → repositories/

❌ 禁止：Module A/routes/ → Module B/routes/
❌ 禁止：直接 import 子路由文件
❌ 禁止：跨模块循环依赖
```

由 `.dependency-cruiser.json` 在 CI 中自动校验。

## 后果

### 正面
- AI 辅助开发时边界清晰，不会跨域乱改
- 单个模块出 bug 只影响该域，不会波及全局
- 社区贡献者可以专注于单一模块开发
- 每个模块有独立 README，AI 和人类都能快速理解

### 负面
- 模块数量较多（18 个），新开发者需要时间熟悉模块边界
- 模块间通信通过服务层，偶尔产生不必要的依赖

### 后续计划
- ~~`infra/` 模块按子域拆分~~ 已完成：`notification/`、`backup/`、`change-management/`、`config-management/` 已拆分为独立模块
- 考虑引入事件总线替代直接的服务层跨模块调用
