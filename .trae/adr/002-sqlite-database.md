# ADR-002: SQLite 作为主数据库

**状态**: 已采纳 | **日期**: 2024-Q4 | **决策者**: 项目作者

## 背景

项目需要持久化存储。核心需求：
1. 零运维成本（无外部数据库依赖）
2. 支持嵌入式部署（用户下载即用，无需配置 DB 连接）
3. 单文件备份/迁移简单
4. 个人项目/小团队场景下性能足够

## 决策

选择 **SQLite + better-sqlite3** 作为主数据库。

## 考虑的替代方案

| 方案 | 优点 | 缺点 | 决策 |
|------|------|------|------|
| **SQLite** | 零配置、单文件、嵌入式、备份简单 | 并发写入弱（单写锁）、不适合高并发 OLTP | ✅ 采纳 |
| PostgreSQL | 功能完整、并发强、扩展丰富 | 需要独立安装运维、不适合嵌入式部署 | 未来可选（企业版） |
| MySQL | 生态大、文档多 | 需要独立运维、许可证风险 | 不采纳 |

## 后果

### 正面
- 用户 clone 项目后 `npm install && npm run dev` 即可运行，无需配置数据库
- 数据库是一个 `.db` 文件，备份只需复制文件
- 通过 migration 框架管理 schema 变更，版本可追溯

### 负面
- 并发写入性能受限（每次只能一个写事务）
- 无法水平扩展
- 时序数据（监控指标）存储效率不如专用时序数据库

### 缓解措施
- 使用 WAL 模式提升并发读性能
- 时序监控数据通过 `baseline_metrics` 表独立存储，定期归档
- 架构上通过 Repository 层抽象，未来可替换为 PostgreSQL 而不影响业务代码
- 大型表按子仓储拆分，控制单表大小

## 设计约束

- **禁止**业务代码直接操作 SQLite 连接（`models/database/index.ts`，含 core/maintenance/health 三个子模块），必须通过 Repository 层
- 新增表必须先写 migration 脚本（`models/migrations/vXXX_xxx.ts`），再创建 repository
- 时序监控指标不存入主业务表
